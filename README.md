# 雾港余烬：2D 对话跑团游戏原型

Godot 4.7 可运行垂直切片，包含约 15 分钟剧本、日期与行动点循环、连续/可中断及可重复事件、地点地图、图片与音乐素材配置、带前置锁的可视化事件编辑、双轨五维育成、六槽装备、堆叠背包、D20 检定、回合制战斗、内容管理和保存读取。

项目内置三张默认图片：大地图、敌人立绘和物品图标。对应内容未配置图片或图片失效时，运行时自动使用 `assets/defaults/` 下的默认素材。

每天拥有3个行动点。主动开始地图事件或资源事件消耗行动点；休息推进一天并恢复行动点，同时消耗20点饱腹。饱腹不足会累计生命上限惩罚，旅行口粮可恢复饱腹。可重复资源事件根据D20检定获得1–4份资源。连续事件开始后必须完成至指定结束节点，可中断事件则允许返回地图安排其他行动。

世界状态包含任务/线索和NPC关系。关键事件会更新当前任务、线索和林鸦/老乔/港民关系；旅馆休息负责推进日期、恢复行动点并结算饱腹消耗。商店用于出售换金道具；旅馆用于调配药剂和整理补给；背包只负责查看、食用和装备。旅馆和商店不再是顶部导航按钮，而是作为地图上的设施入口进入。

## 设计思路概括

项目目标不是制作一部写死流程的视觉小说，而是建立一套可以导入剧本、管理内容并反复扩展的2D跑团框架。整体采用“声明式内容＋通用运行器”：剧本、事件、角色、敌人、装备和物品放在数据层，Godot代码只负责执行规则、显示界面和保存状态。

核心体验由四个循环组成：

1. 玩家在平面大地图点击入口标记进入剧情、随机、可重复事件或城镇设施。
2. 事件通过对话、选项、D20检定或战斗改变世界状态。
3. 每日3个行动点限制探索节奏，可重复事件提供口粮、材料和药草。
4. 休息推进日期、恢复行动点并消耗饱腹，资源不足会影响生命上限。

事件并非全部线性。连续事件用于不能中途离开的关键剧情段，开始后必须执行到指定结束节点；可中断事件用于调查、支线和资源获取，玩家可以返回地图改做其他行动。事件锁通过前置事件列表控制剧情和地点解锁，并支持“全部满足”与“任一满足”，同一规则同时约束直接跳转、地点访问、随机事件池和地图入口显示。

大地图是主要游戏入口。剧情事件、随机事件、可重复事件和设施使用不同标记：`◆` 表示剧情，`✦` 表示随机，`↻` 表示可重复，`旅` 表示旅馆，`商` 表示商店。点击标记直接进入对应事件或设施；行动点不足时需要先去旅馆休息。顶部主菜单区域作为总控制台，只保留开始/继续、地图、背包/装备、角色育成和创作工具等控制入口，不显示游戏内状态。

随机内容采用“模板、标签、条件、权重、固定种子”的思路，而不是让不可控文本直接执行游戏逻辑。当前原型先实现按地点触发和可重复资源事件，为后续事件模板库与剧本包导入保留数据接口。

角色成长分为两套五维：叙事检定五维负责剧情判定，战斗五维负责命中、伤害、防御、速度和生命。装备拥有头盔、盔甲、武器、臂甲、鞋子和饰品六个槽位，可以同时修改两套属性。背包使用物品ID与堆叠数量保存，支持装备、消耗品、任务物品和换金道具。

内容生产工具与游戏运行端共享同一数据模型。创作工具目前包含：

- 可视化事件节点预览、剧情/随机事件类型、地点、前置锁、流程类型、重复触发与行动点设置。
- 自定义事件 ID 由作者手动命名；内置事件 ID 只读，保存时会检查空值、重复和非法字符。
- 角色和敌人的双五维、立绘、装备栏、背包物品及数量配置。
- 装备和物品的分类、图片、描述与属性加成配置。
- 大地图背景图、地图音乐、事件发生点、事件背景图和事件音乐配置。
- 事件文案 JSON 导入；缺少发生点、背景图或音乐等非文案数据时会标记为未完善事件，暂不可触发。

素材采用可替换和可回退策略。导入图片、音乐会复制到 `user://` 内容目录；大地图、敌人和物品没有配置图片时使用项目默认图。音乐使用地图与事件两个播放器：事件有独立音乐时暂停地图音乐，事件音乐为空时从原位置继续地图音乐。

事件界面采用上方 2/3 舞台、下方 1/3 文案框布局。舞台显示事件背景图；未配置事件背景图时回退到大地图背景图。有对话角色时右侧显示对方立绘，左侧显示主角立绘；缺少立绘时回退到默认图。文案在底部显示，长文本可点击文案框翻页，显示到最后一页后选项以可点击文案形式出现。

所有关键随机结果、事件访问、资源获取和战斗结果都写入运行日志。存档记录日期、行动点、饱腹、世界变量、已访问事件、连续事件状态、随机资源次数、角色、背包、装备和战斗状态；旧版背包与新增字段在读取时自动迁移。

## 长线事件扩展与难度体系

本次内容扩展在原型主线之外追加了两大区域、40组连续编号事件，共200个场景节点：`rural_001` 至 `rural_020` 为槐树镇篇，`city_021` 至 `city_040` 为北都篇。每组固定包含5个节点，分别承担开场、交锋、独查、危局和收束功能。重要人物交流场景会写出主角、证人和对立方的立场碰撞；适合主角单独行动的节点则以独查、物证整理和内心推演为主，避免所有场景都被处理成同一种对话节奏。

事件顺序采用严格时间链：槐树镇1号至20号依次解锁，完成 `rural_020_05` 后才进入北都21号事件，北都21号至40号继续线性递进。这样支线虽然仍用于衬托世界观，但不会在时间线上散开成无序素材；每组都能承接上一组留下的证据、人物代价或社会矛盾，并把乡镇的白封条、矿路、义仓、暗账逐步推向北都的救济总署、电报局、议院和钟楼。

难度曲线围绕D20检定和角色成长设计。40组事件的第2节点均设置一次D20检定，难度从11逐步提升到20；槐树镇篇最高难度15，北都篇从16起步，体现区域压力递增。每组第5节点结算完成奖励，提供下一阶段所需的叙事五维成长；包含正面对抗的组还额外提供战斗五维成长、补给、装备或证物。按全流程模拟，主角通过事件完成奖励和育成系统可以将叙事五维稳定推高到足以应对后续难度，同时战斗属性也会随战斗线逐步补强。

新增机制还包括18个可触发战斗事件、40名重要角色参数、5类新增敌人，以及用于奖励和战斗配置的新物品/装备。事件选项可通过 `quest`、`relation`、`coins`、`item_quantity`、`stat_reward` 和 `stat_rewards` 结算任务线索、好感度、银币、堆叽物品和五维属性奖励；`start_battle` 可指定敌人、战斗开场文本和胜利后的后续事件。

## 启动

推荐使用仓库根目录的一键启动脚本：

```powershell
.\launch_game.bat
```

脚本不会写死 Godot 路径。可任选一种方式提供本机 Godot 可执行文件：

- 在仓库根目录创建被 `.gitignore` 忽略的 `godot_path.local.txt`，内容写本机 `Godot_v4.7-stable_win64.exe` 或控制台 exe 的完整路径。
- 设置环境变量 `GODOT_EXE`。
- 将 Godot 可执行文件加入系统 `PATH`。

也可以手动运行，路径按本机情况替换：

```powershell
& '<Godot可执行文件>' --path '<项目目录>'
```

## 内容位置

- `data/scenario.json`：219个带地点、锁、流程、行动点、检定、战斗和音乐属性的事件节点。
- `data/database.json`：带双五维、装备和背包设定的角色/敌人，以及装备、消耗品、任务物品和换金道具。
- `data/event_import_template.json`：事件文案导入模板，包含必填字段、可选素材字段和选项示例。
- `scripts/main.gd`：UI、事件运行器、育成、战斗及创作工具。
- `scripts/game_state.gd`：运行状态、内容库和存档。
- `scripts/content_library.gd`：内容加载、内置/自定义合并、事件表、实体与物品查找。
- `scripts/content_schema.gd`：内容默认值、schema 约束和跨表引用校验。
- `scripts/event_action_schema.gd`：事件选项 action 的标签、字段定义和编辑器表单约束。
- `scripts/scenario_package.gd`：剧本包 JSON 导入、导出与导入前校验。
- `scripts/dice.gd`：D20 与育成骰子。
- `launch_game.bat`：本机一键启动脚本，通过本地配置、环境变量或 PATH 查找 Godot。

存档、自定义内容和导入图片写入 Godot 的 `user://` 目录。事件、角色/敌人、装备/物品和大地图均可选择图片并在编辑器中预览；图片会复制到内容素材目录，内置对象可通过同ID用户配置覆盖。事件发生点保存在自定义内容中，可新增点位或用同名点覆盖内置地点坐标。事件锁支持多个前置事件以及“全部满足/任一满足”，同时约束直接跳转、随机事件池和地点解锁。角色/敌人管理页可设定检定五维、战斗五维、六槽装备及携带物品数量；装备/物品拥有独立管理页。

声音素材支持 MP3、OGG 和 WAV。大地图音乐使用独立循环播放器；事件配置独立音乐时会暂停地图音乐，事件音乐为空时会停止事件通道并从原播放位置继续地图音乐。导入音频会复制到 `user://mist_harbor_assets/audio/`。

创作工具的“内容包概览”页提供剧本包导入/导出、事件文案导入和事件导入模板导出。剧本包是 JSON 文件，包含 `scenario`、`database` 和 `custom_content`；导入前会检查 JSON 结构、事件 ID、跳转目标、D20 分支、前置条件、角色/敌人装备与背包引用。事件文案导入使用轻量 JSON，顶层为 `events` 数组，至少需要 `id`、`title` 和 `text`；`location`、`background_image`、`music` 可后续补齐。自定义内容可用同 ID 覆盖内置事件或条目，校验会将覆盖提示为警告而非错误。

## 事件 action 协议

事件选项统一使用 `text` 加可选效果字段。通用字段包括 `flag`、`item`、`item_quantity`、`damage`、`quest`、`relation`、`coins`、`stat_reward` 和 `stat_rewards`。不设置 `action` 时可用 `next` 跳转到另一个事件；设置 `action` 时支持：

- `open_map`：返回地图。
- `open_growth`：打开角色育成。
- `d20_check`：需要 `stat`、`difficulty`、`success`、`failure`。
- `resource_check`：需要 `stat`、`difficulty`、`resource`。
- `start_battle`：进入战斗，可选 `enemy`、`victory` 和 `battle_intro` 指定敌人、胜利后跳转和开场战斗记录。
- `summary`：进入结算页。

创作工具的事件编辑器已提供选项列表、action 下拉和对应字段表单；保存前会按同一套 schema 生成结构化选项，导入剧本包时也会复用这套校验。
